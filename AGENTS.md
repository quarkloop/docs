# Agent Guide

This is the Quark Docs portal — a Fuma Docs (Next.js) site that aggregates markdown from three product repositories into one searchable, navigable docs site. Your job is to maintain the Next.js app, the sync script, and the build pipeline — not to edit product documentation (that lives in each product repo).

## Repository

- **Name**: Quark Docs Portal
- **Language**: TypeScript (Next.js 15 + React 19)
- **License**: Apache 2.0
- **Repo**: [github.com/quarkloop/docs](https://github.com/quarkloop/docs)
- **Guidelines**: [quarkloop/guidelines](https://github.com/quarkloop/guidelines)

## Quick reference

```bash
# Install dependencies
npm install

# Sync content from product repos (clones quark, quark-js, agent with --depth 1)
./scripts/sync-content.sh

# Start dev server (runs sync first, then next dev)
npm run dev

# Production build (runs sync via vercel.ts buildCommand, then next build, then pagefind)
npm run build

# Type-check
npx tsc --noEmit

# Deploy to Vercel (manual — no CI/CD)
vercel --prod
```

## Rules

### Content management

1. Do not commit `content/` — it is generated by `sync-content.sh` at build time and gitignored.
2. Do not commit `public/pagefind/` — it is generated by the `postbuild` script and gitignored.
3. Do not commit `.source/` — it is generated by Fuma Docs MDX at build time and gitignored.
4. Do not edit product documentation here — edit it in the product repo. This repo only contains the docs site infrastructure.
5. Do not add new product repos to `sync-content.sh` without updating the nav, home page, and docs index to include the new section.
6. Do not hardcode doc filenames in `sync-content.sh` — the script copies ALL `.md` and `.mdx` files from each repo's `docs/` directory.

### Build pipeline

7. The `vercel.ts` `buildCommand` must run `sync-content.sh` before `npm run build` — content must be synced before Next.js compiles.
8. The `postbuild` script must run `pagefind` after `next build` — the search index is generated from the compiled HTML.
9. Do not add a CI/CD workflow — deployment is manual via `vercel --prod`.
10. Do not change the `framework` in `vercel.ts` from `nextjs`.

### Code style

11. Do not disable TypeScript strict mode.
12. Do not use CommonJS — this is an ESM project (`"type": "module"` in package.json).
13. One responsibility per component — if a component grows past 100 lines, split it.
14. Do not add new top-level dependencies without discussing in an issue first.
15. Comments explain **why**, not **what**.

### Design system

16. Do not use gradients — the design system uses solid colors only (sand + ember palette).
17. Do not use cold grays — all colors have a warm undertone (hue 25–35 range).
18. Do not change the `tailwind.config.ts` color palette without discussing — it's the org-wide design language.
19. Use `font-display` class for headings, `font-sans` for body, `font-mono` for code.

## Boundaries

This repo has a clear boundary between **infrastructure** and **content**:

- **Infrastructure** (this repo): `app/`, `components/`, `lib/`, `scripts/`, `vercel.ts`, config files. You edit these.
- **Content** (product repos): `content/docs/platform/`, `content/docs/sdk/`, `content/docs/agent/`. You do NOT edit these — they are synced from the product repos at build time.

The sync script is the bridge:
```
Product repos → sync-content.sh → content/docs/<section>/ → next build → Vercel
```

If a product repo's docs are broken (MDX syntax error, missing frontmatter), the build will fail. Fix the issue in the product repo, not here.

## Commit conventions

- Format: `type(scope): short summary`
- Types: `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `build`, `style`
- Scopes: `app`, `components`, `lib`, `scripts`, `vercel`, `config`, `readme`, `agents`
- One scope per commit — do not mix unrelated changes.
- Inspect staged files before every commit: `git diff --cached --stat`
- Never stage: `content/`, `public/pagefind/`, `.source/`, `node_modules/`, `.next/`, `.vercel/`, `.env`, editor configs.
- Do not include `Co-authored-by` trailers.
- Examples:
  - `feat(app): add dark mode toggle to nav`
  - `fix(scripts): handle missing docs/ directory in product repo`
  - `build(vercel): add cache headers for pagefind index`
  - `style(components): warm-tinted shadows for doc cards`

## Testing

- **Build**: `npm run build` must complete with zero errors. The build runs `sync-content.sh` → `next build` → `pagefind`.
- **Type-check**: `npx tsc --noEmit` must pass with zero errors.
- **Visual**: After build, check that:
  - Home page shows 3 product cards (Platform, SDK, Agent)
  - `/docs` index shows 3 section cards
  - Each section's sidebar lists its pages correctly
  - Pagefind search returns results for common terms
- **Sync**: After running `sync-content.sh`, verify `content/docs/` has 3 subdirectories with `.mdx` files.
- No automated test suite yet (planned — Playwright for visual regression).

## Common mistakes to avoid

1. **Committing `content/`.** The `content/` directory is gitignored for a reason — it's regenerated on every build from the product repos. If you commit it, you'll have stale content that conflicts with the sync script.

2. **Editing product docs here.** If you need to fix a typo in `content/docs/sdk/api.mdx`, edit it in the `quark-js` repo, not here. The change will appear in the docs portal on the next sync.

3. **Forgetting to run sync before dev.** `npm run dev` runs sync automatically, but if you run `next dev` directly, you'll get stale content. Always use `npm run dev`.

4. **Hardcoding filenames in sync-content.sh.** The script copies ALL `.md`/`.mdx` files from each repo's `docs/` directory. Do not add per-file logic — it's not portable and increases maintenance.

5. **Breaking the `vercel.ts` build command.** The `buildCommand` must run sync → build → postbuild (pagefind). If you change the order or remove a step, the build will fail or the search index will be missing.

6. **Using cold grays in the design.** The design system uses warm sand/ember colors. Using `zinc`, `slate`, or `gray` Tailwind colors will break the visual consistency.

7. **Adding CI/CD.** Deployment is manual via `vercel --prod`. Do not add GitHub Actions workflows for deployment — the `vercel.ts` file handles everything.

8. **Ignoring MDX syntax errors.** MDX is stricter than plain Markdown. `<` followed by a digit is parsed as a JSX tag. Use `&lt;` in table cells and prose. Bare `{` and `}` are parsed as JSX expressions. Use `\{` and `\}` or wrap in code blocks.

## Deployment

Deployment is manual via the Vercel CLI. There is no CI/CD integration.

```bash
# Install Vercel CLI (one-time)
npm i -g vercel

# Preview deployment (creates a unique URL)
vercel

# Production deployment (updates the production domain)
vercel --prod
```

The `vercel.ts` file configures:
- **framework**: `nextjs`
- **buildCommand**: `./scripts/sync-content.sh && npm run build` (sync → build → pagefind)
- **cleanUrls**: `true` (no `.html` extensions)
- **trailingSlash**: `false` (prevents duplicate-content SEO issues)
- **headers**: security headers (X-Content-Type-Options, X-Frame-Options, etc.) + cache headers (1-year for static assets, 1-hour for pagefind index)

The sync script uses HTTPS URLs to clone product repos — all three repos are public, so no authentication is needed on Vercel or in CI.

## Sync script internals

`scripts/sync-content.sh` does the following:

1. Cleans `content/docs/` (removes previous sync output)
2. Clones each product repo with `--depth 1` (shallow, no history) to `/tmp/quark-docs-sync/`
3. Copies ALL `.md` and `.mdx` files from each repo's `docs/` directory to `content/docs/<section>/`
4. Lowercases filenames for URL consistency (e.g., `ARCHITECTURE.mdx` → `architecture.mdx`)
5. Copies `meta.json` if present (for sidebar organization); generates one if missing
6. Injects frontmatter (title, description) into files that lack it
7. Generates a top-level `meta.json` listing all sections
8. Cleans up temporary clones

The script does NOT hardcode any filenames. Adding a new doc to any product repo's `docs/` directory automatically includes it in the next sync.
